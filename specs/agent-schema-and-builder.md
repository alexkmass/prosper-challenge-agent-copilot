# Agent Schema & Builder

**Status:** Implemented. Source of truth for the agent data contract â€” update this spec before
changing `backend/agent_builder/`.

## Overview

An agent is a declarative graph: a JSON document (`AgentConfig`) describing nodes (conversation
steps) and edges (LLM-callable transitions between them). `AgentBuilder` loads that JSON, validates
it, and compiles it into a runnable Pipecat Flows graph. This is the one contract every other part of
the system reads and writes â€” the human-facing UI, the Copilot's LLM output, and the voice runtime
all produce or consume the exact same shape. There is no separate "AI format."

## Goals

- One declarative contract, editable by hand, by the graph UI, or generated by an LLM, with no
  translation layer between them.
- Fail loudly at **load time** on a structurally invalid graph, rather than fail silently or crash
  mid-call.
- Stay close to Pipecat Flows' own vocabulary (`role_message`, `task_messages`, `pre/post_actions`)
  so the compiler is a thin, low-risk layer, not a second graph engine.

## Non-goals

- Runtime mutation of the graph mid-call. A call always runs whichever `AgentConfig` was active at
  connection time (see [voice-agent-builder-ui.md](voice-agent-builder-ui.md) for how "active" is
  resolved).
- Semantic/quality validation (tone consistency, whether a task message is well-written,
  missing caller intents). The **Validate** feature's LLM design-review layer handles that
  (see [agent-copilot.md](agent-copilot.md)); this spec covers only the structural contract and
  compile-time rules.
- Node position/layout data. Never part of this contract â€” layout is a UI-only, in-memory concern.

## Requirements

### Data contract

```
AgentConfig:
  name: str
  initial_node: str            # must name a node below
  persona: str = ""            # global role_message, applied to every node unless overridden
  voice_id: str = "21m00Tcm4TlvDq8ikWAM"  # ElevenLabs "Rachel"
  model: str = "gpt-4o"
  nodes: list[Node]

Node:
  name: str
  task_messages: list[{role: str, content: str}] = []   # this node's objective(s)
  role_message: str | None = None                        # overrides persona for this node only
  edges: list[Edge] = []
  pre_actions: list[dict] = []
  post_actions: list[dict] = []
  end: bool = False             # terminal â€” ends the call (mutually exclusive with edges, see below)

Edge:
  function: str                 # tool name the LLM calls to take this transition
  description: str              # tells the model *when* to call it
  target: str                   # must name a node above
  properties: dict = {}         # JSON-schema properties for fields to collect on this edge
  required: list[str] = []      # which of `properties` must be filled before the edge fires
  tool: str | None = None       # optional key into the tool registry, see agent-tools.md
  tool_async: bool = False      # fire-and-forget `tool`, see agent-tools.md
```

Unknown/missing optional fields default as shown; `name`/`initial_node`/`nodes` (on `AgentConfig`)
and `name`/`function`/`description`/`target` (on `Node`/`Edge`) are required keys.

### Compilation behavior

- `AgentBuilder.build_initial_node()` compiles only the entry node; every other node is compiled
  lazily, the moment an edge transitions into it (`_make_node`, called from the edge's own handler).
- A node's effective `role_message` is its own `role_message` if set, else the agent's `persona`.
- Each `Edge` becomes one LLM-callable tool (`FlowsFunctionSchema`): `name=function`,
  `description=description`, JSON-schema `properties`/`required` passed through unchanged.
- Taking an edge: merges the collected `args` into `flow_manager.state` (so later nodes can read
  earlier answers), fires an optional `on_transition(function, target, collected)` callback (used by
  the call log â€” see [voice-agent-builder-ui.md](voice-agent-builder-ui.md)), then compiles and
  returns the target node.
- `post_actions`: explicit `post_actions` on a node always win. Otherwise, if `end` is true, the
  compiled node gets `post_actions: [{"type": "end_conversation"}]` â€” added automatically, not
  something a node author writes by hand.
- `pre_actions` are passed through unchanged when present; omitted entirely when empty (not emitted
  as `[]`).
- If an edge's `tool` is set, its handler awaits that tool's registry entry
  (`handler(args, state)`, see [agent-tools.md](agent-tools.md)) before transitioning, and merges
  the tool's result dict into both the function's return value and `flow_manager.state` alongside
  the collected `args` â€” the edge still always transitions to its own `target`; only the result the
  LLM sees changes.
- Every compiled node whose `end` is `False` also gets two functions not present in
  `AgentConfig` at all: `request_human_agent` and `confirm_human_transfer`, letting a caller
  escalate to a human from anywhere in the graph. See [agent-tools.md](agent-tools.md) for their
  behavior â€” they're a builder-level concern, not something an agent author ever edits.

### Validation rules

Enforced in `backend/agent_builder/validation.py`'s `validate_agent()` â€” a single source of truth
for every structural check. `AgentBuilder._validate()` filters that list to `severity == "error"`
and raises on the first one, so a bad agent never reaches the point of compiling a node. The Validate
feature (`POST /api/copilot/validate`, see [agent-copilot.md](agent-copilot.md)) returns the full
list â€” errors *and* warnings â€” plus an LLM design-review pass on top.

Compile-time **errors** (also block save and generation):
- **FR-1**: the agent must have at least one node.
- **FR-2**: `initial_node` must name a node that exists.
- **FR-3**: every edge's `target` must name a node that exists (no dangling transitions).
- **FR-4**: a node with `end: true` must not also have edges. The two are mutually exclusive, not
  just redundant â€” `end` attaches `end_conversation` as a post-action, which Pipecat Flows executes
  the moment the node is entered (`respond_immediately` defaults to `true`), before any of that
  node's own edges could ever be taken by the caller. (This exact contradiction â€” found live â€”
  crashed a real test call the instant it reached the affected node.)
- **FR-5**: edge `function` names must be unique **within a node**. Two edges on the same node
  sharing a name means the LLM has no way to tell them apart when the model itself picks which tool
  to call; global uniqueness across different nodes is fine since only one node's tools are active at
  a time.
- **FR-6**: an edge's `tool`, if set, must name a key in the tool registry (`TOOL_REGISTRY`, see
  [agent-tools.md](agent-tools.md)) â€” an unknown key is rejected rather than surfacing as an error
  only when the edge is actually called mid-conversation.
- **FR-7**: no edge may use `request_human_agent` or `confirm_human_transfer` as its `function` â€”
  these names are reserved for the two functions auto-injected into every non-terminal node (see
  [agent-tools.md](agent-tools.md)) and a collision would break the model's ability to tell them
  apart.
- **FR-8**: an edge with `tool_async: true` but no `tool` is rejected â€” the flag only means
  something alongside a tool (see [agent-tools.md](agent-tools.md) for what it changes about
  execution; it has no effect on this schema's validation beyond requiring `tool`).

**Warnings** (reported by Validate and `validate_agent()`, but do not block compile/save):

- Self-loop edges (`target` names the same node the edge leaves).
- Unreachable nodes, dead-end non-terminal nodes, no reachable terminal node, start node that is
  terminal, empty task messages, required collect fields not declared in `properties`.

Self-loops are intentionally warnings, not compile errors â€” a self-loop graph can still run, but it
is almost always a design mistake; Copilot prompts forbid them and Validate flags them.
## Acceptance Criteria

All covered by `backend/tests/test_agent_builder.py` (fast, deterministic, no API calls):

- [x] A valid agent (`example_flow2.json`) compiles without error and exposes the expected node set.
- [x] An agent with no nodes is rejected (FR-1).
- [x] An `initial_node` that doesn't exist is rejected (FR-2).
- [x] An edge targeting a nonexistent node is rejected (FR-3).
- [x] A node with `end: true` and non-empty `edges` is rejected (FR-4).
- [x] Two edges on the same node sharing a `function` name are rejected (FR-5).
- [x] The initial node's compiled `functions` list matches its edges' function names exactly.
- [x] A terminal node (`end: true`, no edges) compiles with `post_actions: [{"type": "end_conversation"}]` and an empty `functions` list.
- [x] An edge with an unknown `tool` key is rejected (FR-6).
- [x] An edge named `request_human_agent` or `confirm_human_transfer` is rejected (FR-7).
- [x] An edge with `tool_async: true` and no `tool` is rejected (FR-8).
- [x] A non-terminal node's compiled `functions` includes `request_human_agent` and
      `confirm_human_transfer` in addition to its own edges; a terminal node's does not.

## Out of Scope / Deferred

- No JSON Schema / OpenAPI-level schema validation beyond what `AgentBuilder._validate()` checks â€”
  malformed `properties` dicts (e.g. an invalid JSON-schema `type`) aren't currently caught here; they
  would surface as an OpenAI API error at call time instead.
- No versioning/migration story for the `AgentConfig` shape. Any field rename would need every stored
  agent in `AgentStore` updated by hand (acceptable for a single-process, in-memory dev tool).
